## 1. 개요

Tower Breakers의 적 군집 이동 시스템은 **단일 연결 리스트(Singly Linked List) 기반의 기차 대열(Train Formation)** 알고리즘으로 구현되어 있다. 수십 마리의 적이 일렬로 플레이어를 향해 전진하면서, 물리적으로 겹치지 않고 자연스러운 압박감을 제공하는 것이 핵심 목표이다.

### 1.1 설계 목표

| 목표 | 설명 |
|------|------|
| **대열 유지** | N마리의 적이 일정 간격(`TrainSpacing`)을 유지하며 한 줄로 전진 |
| **전파식 정지** | 선두가 플레이어/벽에 막히면 뒤따르는 적도 순차적으로 정지 |
| **플레이어 밀기** | 선두 적이 플레이어를 접촉 시 왼쪽으로 밀어냄 (Push Force) |
| **사망 시 대열 재연결** | 중간 적 사망 시 앞뒤 노드가 자동으로 연결됨 |
| **Zero Allocation** | Update 루프 내 힙 할당 없이 동작 |

## 2. 자료구조: 단일 연결 리스트

```
[리더] ← [2번] ← [3번] ← ... ← [꼬리]
 (m_aheadEnemy = null)    (m_aheadEnemy = 앞 노드)
 (m_followerEnemy = 2번)  (m_followerEnemy = 뒤 노드 or null)
```

### 2.1 노드 구조 (`EnemyPushLogic`)

```csharp
// 이중 포인터로 양방향 탐색 가능
private EnemyPushLogic m_aheadEnemy;     // 내 앞의 적 (플레이어 방향)
private EnemyPushLogic m_followerEnemy;  // 내 뒤의 적 (스폰 방향)
```

- **리더(Leader)**: `m_aheadEnemy == null`인 최전방 노드
- **꼬리(Tail)**: `m_followerEnemy == null`인 최후방 노드

### 2.2 대열 연결 (`EnemySpawner.SpawnFloorEnemiesAsync`)

스폰 시 `previousEnemy` 변수로 이전 적을 추적하며, 새로운 적이 생성될 때마다 연결:

```csharp
logic.TrainSpacing = floorData.TrainSpacing;
logic.SetAheadEnemy(previousEnemy);   // 이전 적을 내 앞의 적으로 설정
previousEnemy = logic;                // 현재 적을 다음 적의 "앞"으로 기억
```

`SetAheadEnemy` 호출 시 양방향 포인터가 동시에 갱신된다:

```csharp
public void SetAheadEnemy(EnemyPushLogic ahead)
{
    m_aheadEnemy = ahead;
    if (m_aheadEnemy != null)
        m_aheadEnemy.SetFollower(this);  // 역방향 포인터 자동 갱신
    InvalidateLeaderCache();
    UpdateCollider();
}
```

### 2.3 사망 시 대열 재연결 (`HandleDeath`)

중간 노드가 제거되면 앞뒤 노드를 직접 연결하여 리스트 무결성을 유지한다:

```
[A] ← [B(사망)] ← [C]
        ↓ HandleDeath()
[A] ← [C]
```

```csharp
public void HandleDeath()
{
    if (m_followerEnemy != null)
        m_followerEnemy.SetAheadEnemy(m_aheadEnemy);  // C의 ahead를 A로
    else if (m_aheadEnemy != null)
        m_aheadEnemy.SetFollower(null);                // A의 follower를 null로

    m_aheadEnemy = null;
    m_followerEnemy = null;
    InvalidateLeaderCache();
    UnregisterEnemy(this);
}
```

## 3. 이동 알고리즘

### 3.1 프레임별 이동 흐름 (`EnemyMovementHelper.ExecuteMovement`)

```
매 프레임 (OnTick)
├── 1. 차단 상태 확인: pushLogic.IsBlocked(gap)
├── 2. 애니메이션 전환: MOVE ↔ IDLE (상태 변경 시에만)
├── 3. 이동 수행 (미차단 시)
│   ├── pos.x -= moveDelta
│   ├── 앞 적과 최소 간격 강제 유지 (pos.x < minX → pos.x = minX)
│   └── 플레이어 접촉 시 SyncPositionDelta(-moveDelta)
└── 4. TryPushPlayer(): 유효 사거리 내 힘 전달
```

### 3.2 차단 판정 (전파식 정지)

`IsBlocked(gap)` 메서드는 **재귀적 전파 + 프레임 캐싱** 방식으로 동작한다:

```
[리더] ← [2번] ← [3번]

리더.IsBlocked(gap):
  ├── 리더가 플레이어와 닿음 + 플레이어가 벽에 있음 → blocked = true
  └── 캐시에 저장 (m_lastBlockedFrame = Time.frameCount)

2번.IsBlocked(gap):
  ├── 앞 적(리더)과 간격 ≤ TrainSpacing + 0.1
  ├── 리더.IsBlocked(gap) → 캐시 즉시 반환 (true)
  └── blocked = true → 캐시에 저장

3번.IsBlocked(gap):
  ├── 앞 적(2번)과 간격 ≤ TrainSpacing + 0.1
  ├── 2번.IsBlocked(gap) → 캐시 즉시 반환 (true)
  └── blocked = true → 캐시에 저장
```

차단 조건 요약:

| 조건 | 결과 |
|------|------|
| 플레이어 접촉 + 플레이어 벽 충돌 | 차단 |
| 앞 적이 차단 + 간격 ≤ `TrainSpacing + 0.1` | 차단 (전파) |
| 위 조건 모두 해당 없음 | 이동 가능 |

### 3.3 간격 유지 (Anti-Overlap)

이동 처리 시 앞 적을 앞지르지 못하도록 하드 클램핑:

```csharp
if (pushLogic.AheadEnemy != null)
{
    float minX = pushLogic.AheadEnemy.transform.position.x + pushLogic.TrainSpacing;
    if (pos.x < minX)
    {
        pos.x = minX;  // 강제 간격 유지
    }
}
```

### 3.4 플레이어 밀기 메커니즘

두 가지 경로로 플레이어에게 힘을 전달한다:

1. **위치 동기화** (`SyncPositionDelta`): 적이 전진할 때 동일 변위를 플레이어에게 적용
2. **힘 기반 밀기** (`ApplyPushForce`): `deltaTime` 기반 연속 힘 전달

```
적 전진 (moveDelta)
  ├── 플레이어 접촉 시 → SyncPositionDelta(-moveDelta)
  └── TryPushPlayer() → 유효 사거리(±1.0~1.2) 내 ApplyPushForce(m_force)
```

플레이어 측(`PlayerPushReceiver`)에서 저항 계수를 적용:

```csharp
float resistance = m_model.FinalPushResistance;    // 장비 보정 포함
float adjustedDelta = deltaX * (1.0f - resistance);
```

## 4. 최적화 기법

### 4.1 콜라이더 선택적 활성화

N마리 중 **리더만 Collider2D를 활성화**하여 물리 연산량을 `O(N)` → `O(1)`로 절감:

```csharp
public void UpdateCollider()
{
    col.enabled = m_isSpecialType || (m_aheadEnemy == null);
    // 리더(aheadEnemy == null)이거나 특수 개체만 활성화
}
```

스킬 사용 시에는 전체 콜라이더를 일시적으로 활성화/비활성화:

```csharp
EnableAllGroupCollidersForSkill();   // 스킬 시작: 전원 활성화 (타격 판정용)
DisableAllGroupCollidersForSkill();  // 스킬 종료: 리더만 복귀
```

### 4.2 리더 캐싱 (세대 번호 패턴)

`GetLeader()`는 연결 리스트를 순회하여 리더를 찾는 `O(N)` 연산이다. 이를 **세대 번호(Generation)** 기반 캐싱으로 최적화:

```csharp
private static int s_generation = 0;         // 전역 세대 번호
private EnemyPushLogic m_cachedLeader;        // 캐시된 리더
private int m_cachedLeaderGen = -1;           // 캐시 시점의 세대 번호

public EnemyPushLogic GetLeader()
{
    // 세대 번호 일치 → O(1) 즉시 반환
    if (m_cachedLeaderGen == s_generation && m_cachedLeader != null)
        return m_cachedLeader;

    // 세대 불일치 → O(N) 순회 후 캐시 갱신
    EnemyPushLogic current = this;
    while (current.m_aheadEnemy != null && current.m_aheadEnemy.gameObject.activeInHierarchy)
        current = current.m_aheadEnemy;

    m_cachedLeader = current;
    m_cachedLeaderGen = s_generation;
    return current;
}
```

대열 변경 이벤트(`SetAheadEnemy`, `HandleDeath`) 시 `s_generation++`로 모든 캐시를 일괄 무효화:

```csharp
public static void InvalidateLeaderCache()
{
    s_generation++;  // 모든 적의 캐시가 한 번에 무효화됨
}
```

### 4.3 차단 판정 프레임 캐싱

`IsBlocked(gap)`는 재귀 호출로 `O(N²)` 가능성이 있다. **프레임 단위 캐싱**으로 `O(N)`으로 개선:

```csharp
private bool m_cachedBlocked;
private int m_lastBlockedFrame = -1;

public bool IsBlocked(float gap)
{
    // 같은 프레임이면 캐시 즉시 반환 → 중복 재귀 차단
    if (m_lastBlockedFrame == Time.frameCount)
        return m_cachedBlocked;

    // ... 판정 로직 ...

    m_cachedBlocked = isBlocked;
    m_lastBlockedFrame = Time.frameCount;
    return isBlocked;
}
```

마찬가지로 `IsGroupBlocked(gap)`도 리더 단위로 프레임 캐싱:

```csharp
if (leader.m_cachedBlockedGen == Time.frameCount)
    return leader.m_cachedGroupBlocked;  // 같은 프레임 → 캐시 반환
```

### 4.4 정적 레지스트리 (FindObjectsOfType 대체)

런타임 `FindObjectsOfType<EnemyPushLogic>()` 호출을 제거하고, 정적 리스트로 관리:

```csharp
private static readonly List<EnemyPushLogic> s_activeEnemies = new();

public static void RegisterEnemy(EnemyPushLogic enemy) { ... }
public static void UnregisterEnemy(EnemyPushLogic enemy) { ... }
```

- **등록 시점**: `ResetState()` (풀링 재사용 시)
- **해제 시점**: `HandleDeath()` (사망 시)
- **접근**: `EnemyPushLogic.ActiveEnemies` (읽기 전용)

### 4.5 Transform 참조 캐싱

매 프레임 `transform` 프로퍼티 접근을 줄이기 위해 초기화 시 캐싱:

```csharp
private Transform m_cachedTransform;      // 자신의 Transform
private Transform m_targetTransform;      // 플레이어 Transform

public void Initialize(float force, PlayerPushReceiver target, bool isSpecial)
{
    m_cachedTransform = transform;
    m_targetTransform = m_target.transform;
}
```

## 7. 시간 복잡도 요약

| 연산 | 최적화 전 | 최적화 후 | 기법 |
|------|----------|----------|------|
| `GetLeader()` | O(N) 매 호출 | O(1) 평균 | 세대 번호 캐싱 |
| `IsBlocked()` | O(N²) 최악 | O(N) 최악, O(1) 평균 | 프레임 캐싱 |
| `IsGroupBlocked()` | O(N) 매 호출 | O(1) 평균 | 리더 프레임 캐싱 |
| 콜라이더 물리 연산 | O(N) 매 프레임 | O(1) 평시, O(N) 스킬 시 | 리더 전용 활성화 |
| 활성 적 조회 | O(씬 전체) | O(1) | 정적 레지스트리 |
| HandleDeath 대열 재연결 | O(1) | O(1) | 양방향 포인터 |

## 8. 스폰 전략

### 8.1 단일 패킷 (적 타입 1종)

순차 스폰 + 고정 간격(`TrainSpacing`) + 패킷 간 여유분 2.0 유닛 추가.

### 8.2 복수 패킷 (적 타입 2종+)

**Fisher-Yates Shuffle**로 무작위 섞어서 스폰:

```csharp
for (int i = m_spawnMixedPool.Count - 1; i > 0; i--)
{
    int rnd = Random.Range(0, i + 1);
    (m_spawnMixedPool[i], m_spawnMixedPool[rnd]) = (m_spawnMixedPool[rnd], m_spawnMixedPool[i]);
}
```

- GC 방지: `m_spawnMixedPool`은 클래스 필드로 재사용
- 결과: 동일 층에 Normal + Elite 등이 자연스럽게 혼합 배치됨

## 9. 메모리 할당 분석 (Update 루프)

| 항목 | 힙 할당 | 비고 |
|------|---------|------|
| `ExecuteMovement()` | 0 | 모든 연산이 스택/캐시 기반 |
| `IsBlocked()` | 0 | 프레임 캐싱, 재귀이나 힙 할당 없음 |
| `GetLeader()` | 0 | 세대 번호 기반 캐시 |
| `TryPushPlayer()` | 0 | 단순 거리 비교 |
| `SyncPositionDelta()` | 0 | Vector2 스택 할당 |
| 콜라이더 Toggle | 0 | 정적 리스트 순회 |

**결론: Update 루프 내 Zero Allocation 달성.**
